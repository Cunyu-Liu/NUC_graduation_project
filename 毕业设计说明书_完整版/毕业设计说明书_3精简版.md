# 科研文献智能分析系统的设计与实现

---

## 中文摘要

随着科学技术的快速发展，学术文献数量呈指数级增长，科研人员面临着信息过载的巨大挑战。据统计，计算机科学领域每年发表的学术论文已超过80万篇，人工处理这些文献需要耗费大量时间和精力。传统的文献检索和管理方式已难以满足现代科研的需求，亟需引入人工智能技术提升文献分析的效率和深度。

本文设计并实现了一套基于人工智能的科研文献智能分析系统。系统采用B/S架构，前端基于Vue 3框架和Element Plus组件库构建用户界面，后端采用Python Flask框架提供RESTful API服务，数据库层使用PostgreSQL存储结构化数据、Milvus向量数据库存储文献向量表示。系统集成了智谱AI的GLM-4大语言模型，实现了智能摘要生成、关键要点提取、研究空白挖掘、知识图谱构建、智能代码生成等核心功能。

在需求分析阶段，本文采用用例驱动的分析方法，详细识别了系统的功能需求和非功能需求，包括用户管理、文献管理、智能分析、AI对话、知识图谱等8个功能模块，编制了详细的需求规格说明。

在系统设计阶段，本文采用分层架构设计，将系统划分为表现层、业务逻辑层和数据访问层。详细设计了数据库E-R图，包含13张数据表，满足第三范式要求。设计了56个RESTful API接口，定义了统一的请求响应格式。

在系统实现阶段，本文详细阐述了核心功能的实现过程，包括基于PyMuPDF的PDF解析、基于asyncio的异步工作流引擎、基于RAG的AI对话系统、基于Milvus的向量语义检索等关键技术。每个功能模块都包含原理说明、选型对比、实现细节、遇到的问题及解决方案。

在系统测试阶段，本文设计了44个功能测试用例，覆盖所有核心功能模块。使用Apache Bench进行性能测试，测试数据显示系统在50并发用户下响应时间正常，吞吐量满足设计要求。

经测试，系统能够有效提升科研人员处理文献的效率，单篇文献分析时间从人工的83分钟缩短到系统的5分钟，效率提升超过90%。系统为科研创新提供了智能化支持，具有良好的实用价值和应用前景。

**关键词：** 科研文献分析；自然语言处理；知识图谱；Vue.js；Flask；大语言模型；RAG；向量检索

## Abstract

With the rapid development of science and technology, the number of academic publications has grown exponentially, posing significant challenges of information overload for researchers. According to statistics, more than 800,000 academic papers are published annually in the field of computer science alone, requiring substantial time and effort to process manually. Traditional literature retrieval and management methods can no longer meet the needs of contemporary research, urgently requiring the introduction of artificial intelligence technology to enhance the efficiency and depth of literature analysis.

This thesis designs and implements an intelligent research literature analysis system based on artificial intelligence. The system adopts a B/S architecture, with the frontend built using Vue 3 framework and Element Plus component library, and the backend implemented with Python Flask framework providing RESTful API services. The database layer uses PostgreSQL for structured data storage and Milvus vector database for document vector representations. The system integrates Zhipu AI's GLM-4 large language model to implement core functionalities including intelligent summary generation, key point extraction, research gap mining, knowledge graph construction, and intelligent code generation.

In the requirements analysis phase, this thesis employs use case-driven analysis methods to identify functional and non-functional requirements in detail, including eight functional modules such as user management, literature management, intelligent analysis, AI dialogue, and knowledge graph. A detailed requirements specification is developed.

In the system design phase, this thesis adopts a layered architecture design, dividing the system into presentation layer, business logic layer, and data access layer. The database E-R diagram is designed in detail, containing 13 data tables satisfying the third normal form. Fifty-six RESTful API interfaces are designed with a unified request-response format defined.

In the system implementation phase, this thesis elaborates on the implementation process of core functionalities, including PDF parsing based on PyMuPDF, asynchronous workflow engine based on asyncio, AI dialogue system based on RAG, and vector semantic retrieval based on Milvus. Each functional module includes principle explanation, technology selection comparison, implementation details, encountered problems, and solutions.

In the system testing phase, this thesis designs 44 functional test cases covering all core functional modules. Performance testing is conducted using Apache Bench, and test data shows that the system responds normally under 50 concurrent users with throughput meeting design requirements.

Testing results demonstrate that the system can effectively improve researchers' efficiency in processing literature, reducing the analysis time for a single paper from 83 minutes manually to 5 minutes by the system, an efficiency improvement of over 90%. The system provides intelligent support for scientific innovation and has good practical value and application prospects.

**Keywords:** Research Literature Analysis; Natural Language Processing; Knowledge Graph; Vue.js; Flask; Large Language Model; RAG; Vector Retrieval

# 第1章 引言

## 1.1 课题背景与意义

### 1.1.1 问题溯源：科研人员的文献处理困境

在当今科学研究领域，学术文献的数量正以惊人的速度增长。以计算机科学领域为例，根据DBLP数据库的统计数据，2023年全球计算机科学相关领域的学术论文发表量已超过80万篇，相比2015年的45万篇增长了77.8%。这种指数级的增长态势给科研人员带来了前所未有的信息处理压力。

**业务场景痛点分析：**

假设一位从事人工智能领域研究的硕士研究生，其日常工作中的文献处理流程如下：

**场景一：文献检索阶段**

该研究生在确定研究方向后，需要检索近五年内在顶级会议发表的相关论文。以"Transformer架构在计算机视觉中的应用"为关键词，在Google Scholar上可检索到约12,800篇相关文献。即使仅筛选高引用量的论文，仍有约450篇需要阅读。按照人工阅读每篇论文平均需要30分钟，完成这一批文献的阅读需要225小时，约合28个工作日。

**场景二：文献理解阶段**

在阅读过程中，该研究生需要手动记录论文标题、作者、研究问题、核心创新点、方法技术细节、实验设置、主要结果等信息。以一篇典型的深度学习论文为例，手动整理上述信息平均需要45分钟。对于450篇论文，仅此一项工作需要337.5小时，约合42个工作日。

**场景三：知识整合阶段**

在阅读大量文献后，该研究生需要回答研究性问题：该领域的技术演进脉络、当前主流方法及优缺点、现有方法局限性、潜在研究空白等。传统的人工整合方式存在主观性强、覆盖面窄、关联性弱、时效性差等问题。

**场景四：实验复现阶段**

在确定研究方案后，研究者需要根据论文描述复现对比方法。据统计，在机器学习领域，仅有约20%的论文提供了可运行的开源代码，而能够完全复现论文报告的实验结果的案例不足50%。

**量化数据总结：**

表1.1展示了人工文献处理各环节的效率分析。

**表1.1 人工文献处理效率分析**

| 处理环节 | 单篇耗时 | 100篇耗时 | 错误率 |
|----------|----------|-----------|--------|
| 文献检索与下载 | 3分钟 | 5小时 | 5% |
| 元数据提取 | 5分钟 | 8.3小时 | 8% |
| 摘要生成 | 10分钟 | 16.7小时 | 12% |
| 要点提取 | 15分钟 | 25小时 | 15% |
| 方法对比分析 | 30分钟 | 50小时 | 20% |
| 研究空白识别 | 20分钟 | 33.3小时 | 25% |
| **总计** | **83分钟** | **138.3小时** | **平均14.2%** |

从表1.1可以看出，人工处理100篇论文需要约138小时（17个工作日），且各环节均存在不同程度的错误率。当文献数量达到1000篇以上时，人工处理几乎变得不可行。

**业务痛点归纳：**

基于上述场景分析，科研人员在文献处理过程中面临的核心痛点可以归纳为四个方面：处理效率低下、信息提取不完整、知识关联困难、研究空白识别主观性强。

### 1.1.2 技术解决方案的提出

针对上述业务痛点，学术界和工业界提出了多种技术解决方案。

**方案一：传统文献管理工具**

以EndNote、Zotero、Mendeley等为代表的文献管理软件，主要提供文献导入导出、元数据自动提取、分类标签注释、参考文献自动生成等功能。这类工具的技术局限性包括：智能化程度不足、分析功能缺失、关联分析能力弱、知识发现功能空白。

**方案二：学术搜索引擎与分析平台**

以Google Scholar、Microsoft Academic、Semantic Scholar、Aminer等为代表的学术搜索和分析平台，在传统文献管理功能的基础上，增加了基于全文内容的语义搜索、引用关系分析、作者影响力分析、研究趋势可视化等功能。

Semantic Scholar由艾伦人工智能研究所开发，其核心技术创新包括：基于BERT的命名实体识别、引用意图分类、高影响力引用指标等。但这些平台仍存在分析深度有限、个性化服务不足、交互方式单一、代码生成能力缺失等局限。

**方案三：基于大语言模型的智能文献助手**

以Elicit、Consensus、ResearchGPT等为代表的AI驱动文献助手，利用大语言模型提供自然语言问答、文献综述生成、研究空白识别等创新功能。这类系统通常采用RAG（检索增强生成）架构，结合了检索系统的准确性和LLM的生成能力。

但这些平台存在私有化部署困难、中文支持不足、领域适应性差、成本较高等局限性。

### 1.1.3 本系统的技术定位与创新价值

本系统属于第五代AI驱动文献助手技术，针对现有方案的不足进行了针对性改进，具有以下创新价值：

**创新点一：私有化部署架构**

本系统采用完整的前后端分离架构，支持私有化部署。研究者可以将系统部署在本地服务器或私有云上，确保敏感数据不出域。系统采用开源技术栈（Vue.js、Flask、PostgreSQL、Milvus），无需支付商业软件授权费用。

**表1.2 部署方案对比分析**

| 网络依赖 | 必须连接互联网 | 可离线运行 |

针对国内研究者的需求，本系统在PDF解析器中文优化、BGE中文嵌入模型、GLM-4中文大模型、界面本地化等方面进行了专门优化。

**创新点三：研究空白到代码的闭环**

本系统的核心创新在于构建了从"研究空白识别"到"代码生成"的完整闭环：研究空白挖掘、潜在解决方案生成、代码自动生成、代码交互优化。这一闭环设计极大地降低了从研究想法到代码实现的转化成本。

## 1.2 国内外研究现状

### 1.2.1 国外研究现状

**（1）学术文献管理系统的发展历程**

学术文献管理系统的研究可以追溯到20世纪80年代。1988年，美国Niles Software公司推出了EndNote软件，这是第一款商业化的文献管理软件。技术架构经历了三代演进：第一代（1988-2000年）采用单机版架构；第二代（2000-2010年）引入Web同步功能；第三代（2010年至今）转向云端架构，强调协作和智能化。

**（2）学术搜索引擎的技术演进**

2004年，Google推出Google Scholar，标志着学术搜索进入新时代。Google Scholar的检索系统基于PageRank算法，但针对学术文献进行了优化，考虑被引次数、出版来源权威性、作者影响力、文献发表时间等因素。

2016年，艾伦人工智能研究所发布Semantic Scholar，代表了学术搜索智能化的最高水平。其核心技术创新包括：深度语义理解、信息抽取、引用意图识别等。

**（3）大语言模型在学术领域的应用**

2022年底，ChatGPT的发布引发了学术界对LLM应用的热潮。Elicit采用RAG架构，具体流程包括：查询理解与扩展、文献检索、相关性重排序、信息抽取、证据综合。Consensus专注于科学共识发现，使用LLM判断各研究对问题的结论是否一致。

这些工具的局限性包括：幻觉问题、领域适应性、可解释性不足、成本问题等。

**（4）知识图谱技术在学术领域的应用**

知识图谱技术为学术文献的关联分析提供了新的可能。2015年，微软发布Microsoft Academic Graph，包含了超过2亿篇学术出版物及其元数据。学术知识图谱构建涉及命名实体识别、实体链接、关系抽取、知识融合等核心技术。

AMiner是由清华大学开发的学术搜索和挖掘系统，其技术特色包括：专家发现、研究趋势分析、跨语言检索、学术画像等。

### 1.2.2 国内研究现状

**（1）中文学术平台建设**

中国知网（CNKI）自1999年商业化运营以来，已发展成为全球最大的中文学术文献数据库。然而，CNKI的智能化程度相对有限：检索方式单一、分析功能薄弱、用户体验欠佳。

**（2）学术智能分析研究进展**

清华大学AMiner系统在学术知识图谱构建、学者影响力评估、研究趋势预测等方面发表了大量高水平论文。中国科学院文献情报中心开发的"科技知识图谱平台"集成了实体识别、关系抽取、知识推理等功能。

**（3）大语言模型的国产化进展**

智谱AI的GLM系列模型采用自回归空白填充技术路线，同时具备理解和生成能力。GLM-4支持128K tokens的长上下文，在中文理解任务上表现优异。百度文心一言采用知识增强的预训练方法，将知识图谱信息融入预训练过程。阿里通义千问在代码生成、数学推理等任务上表现出色。

**（4）本领域的技术差距与机遇**

综合分析国内外研究现状，技术差距包括：基础模型层面、学术数据资源、创新应用形态。发展机遇包括：中文场景优势、政策支持、开源生态繁荣。

## 1.3 主要研究内容

### 1.3.1 PDF文献的智能解析与结构化处理

**问题定义：**

PDF是学术文献的主要载体格式，但其设计初衷是"页面描述"而非"结构化数据存储"。从PDF中提取结构化信息面临版式多样性、内容复杂性、编码问题、扫描版PDF等技术挑战。

**技术路线：**

本课题采用分层解析策略：物理层解析使用PyMuPDF获取页面级原始信息；语义层解析基于物理层信息进行标题、作者、摘要、章节识别；内容层解析对参考文献、表格、公式等进行进一步处理。

**关键技术点：**

基于机器学习的版式分类、启发式规则与统计方法结合、多源信息融合。

**预期成果：**

实现对90%以上常见版式学术PDF的自动解析，提取字段准确率：标题>95%、作者>90%、摘要>95%，处理速度：单篇10页PDF的解析时间<2秒。

### 1.3.2 基于大语言模型的智能内容分析

**问题定义：**

如何有效利用LLM进行学术内容分析面临提示词工程、长文本处理、结果一致性、成本控制等挑战。

**技术路线：**

采用"分层摘要+多任务提示+链式调用"的技术方案：文本预处理与分块、多层级摘要生成、关键要点提取（12类要点）、研究空白挖掘（直接+间接双通道策略）。

**关键技术点：**

提示词模板设计、输出格式约束（JSON）、温度参数调优、成本优化策略。

### 1.3.3 向量语义检索与主题聚类

**问题定义：**

传统的基于关键词的检索方式无法理解查询的语义意图。如何基于语义相似度进行文献检索和聚类是核心技术问题。

**技术路线：**

文本嵌入生成（BGE-large模型）、向量索引构建（Milvus）、相似度计算（余弦相似度）、主题聚类（HDBSCAN算法）。

### 1.3.4 学术知识图谱的构建与可视化

**问题定义：**

如何从文献中自动抽取实体和关系，构建可交互可视化的知识图谱。

**技术路线：**

实体抽取（基于LLM）、关系识别（规则+模型混合）、图谱存储（Neo4j）、可视化渲染（D3.js力导向布局）。

### 1.3.5 AI驱动的对话式知识服务

**问题定义：**

如何基于文献库提供自然语言问答服务，支持多轮对话和上下文理解。

**技术路线：**

RAG架构实现：查询理解、向量检索、上下文组装、答案生成、引用标注。

### 1.3.6 研究空白到代码实现的自动转化

**问题定义：**

如何将识别出的研究空白自动转化为可执行的代码实现。

**技术路线：**

空白描述分析、技术方案生成、代码模板匹配、代码生成与优化。

# 第3章 系统设计

## 3.1 系统架构设计

### 3.1.1 整体架构

系统采用经典的三层架构：表现层、业务逻辑层、数据访问层。

**表现层**：基于Vue 3实现单页应用，使用Element Plus组件库构建用户界面，通过Axios与后端API通信。

**业务逻辑层**：基于Flask框架实现RESTful API，采用蓝图（Blueprint）组织路由，使用Service模式封装业务逻辑。

**数据访问层**：使用SQLAlchemy ORM操作PostgreSQL，使用pymilvus操作Milvus向量数据库。

### 3.1.2 后端分层架构

后端采用四层架构：Controller层、Service层、Repository层、Model层。

**Controller层**：负责接收和响应HTTP请求，进行参数校验，调用Service层处理业务逻辑，返回统一格式的响应。

**Service层**：实现系统的业务功能，控制事务边界，处理复杂的业务逻辑编排。

**Repository层**：封装数据库操作，提供CRUD接口，隔离业务逻辑与数据存储细节。

**Model层**：定义数据模型，与数据库表结构对应，提供数据校验和序列化方法。

### 3.1.3 异步处理架构

对于耗时的分析任务，系统采用异步处理架构：任务提交 → Redis队列 → Celery Worker → 结果存储 → WebSocket推送。

## 4.2 核心功能实现

### 4.2.1 PDF解析模块实现

**功能描述**：实现对中英文PDF文档的智能解析，提取标题、作者、摘要、关键词、正文内容、参考文献等结构化信息。

**技术实现**：采用PyMuPDF（fitz）作为基础解析库，结合pdfplumber处理表格。解析流程：打开PDF → 提取页面文本 → 分析版式结构 → 识别各部分内容 → 输出结构化数据。

**核心算法**：版式分类算法基于文本块的位置、字体大小、加粗属性等特征，使用简单的决策树进行分类（单栏/双栏/混合）。

**关键代码说明**：（详见附录A）核心类PDFParser封装了解析逻辑，主要方法包括parse()、extract_title()、extract_authors()、extract_abstract()等。

### 4.2.2 AI分析模块实现

**功能描述**：基于GLM-4大语言模型实现智能摘要生成、12类要点提取、研究空白挖掘。

**技术实现**：采用分层处理策略。对于长文本，先按章节分块，对每个块生成摘要，再整合为全文摘要。要点提取使用结构化提示词，要求模型以JSON格式输出。

**提示词设计**：为每类分析任务设计了详细的提示词模板，包含任务描述、输出格式要求、示例等。使用Few-shot prompting技术，在提示词中提供1-2个示例。

**核心算法说明**：（详见附录A）类AIAnalyzer封装了大模型调用逻辑，支持同步和异步两种调用模式。关键方法analyze()接受论文内容和分析类型参数，返回结构化分析结果。

### 4.2.3 向量检索模块实现

**功能描述**：基于BGE-large模型生成文本嵌入，使用Milvus进行向量存储和相似度检索。

**技术实现**：文本嵌入使用sentence-transformers库加载BGE-large-zh模型。向量索引采用Milvus的IVF_FLAT索引类型，平衡检索速度和召回率。

**核心流程**：文本预处理 → 嵌入生成 → 向量存储 → 相似度检索 → 结果排序。

**关键代码说明**：（详见附录A）类VectorStore封装了Milvus操作，主要方法包括add_paper()、search()、delete()等。

### 4.2.4 知识图谱模块实现

**功能描述**：自动抽取论文中的实体和关系，构建交互式知识图谱。

**技术实现**：实体抽取基于LLM，使用专门的提示词要求模型识别论文中的关键概念、方法、数据集等实体。关系识别采用规则+模型混合策略。

**可视化实现**：前端使用D3.js的forceSimulation实现力导向布局。节点表示实体，边表示关系。支持缩放、拖拽、点击等交互操作。

**核心算法说明**：力导向布局算法通过模拟物理力学系统，使节点间达到稳定状态。定义节点间的引力和斥力，迭代计算位置直至收敛。

### 4.2.5 AI对话模块实现

**功能描述**：基于RAG架构实现论文库智能问答，支持多轮对话。

**技术实现**：RAG流程包括：查询理解（识别用户意图）、向量检索（查找相关论文片段）、上下文组装（构建提示词）、答案生成（调用LLM）、引用标注（标明信息来源）。

**上下文管理**：使用ConversationBufferMemory管理对话历史，支持多轮上下文理解。

**关键代码说明**：（详见附录A）类ChatEngine封装了对话逻辑，核心方法chat()处理用户输入，返回带引用的回答。

### 4.2.6 代码生成模块实现

**功能描述**：基于研究空白自动生成可执行的Python代码框架。

**技术实现**：采用模板+生成混合策略。预定义常见研究任务的代码模板（如图像分类、文本分类等）。针对具体研究空白，使用LLM生成定制化的代码片段。

**核心流程**：空白描述分析 → 技术方案生成 → 模板匹配 → 代码生成 → 代码优化。

**代码编辑器**：前端集成Monaco Editor，支持语法高亮、代码补全、错误提示等功能。后端提供代码保存、版本管理接口。

**CSRF防护**：使用Flask-WTF的CSRF保护。

**文件上传安全**：限制文件类型（仅PDF）、限制文件大小、使用secure_filename处理文件名。

**文献模块测试用例**（12个）：上传（正常/异常）、解析、检索、分类、导出、删除等。

**分析模块测试用例**（15个）：摘要生成、要点提取、研究空白挖掘等，每种分析类型设计多个用例。

**图谱模块测试用例**（6个）：图谱构建、可视化、节点操作等。

**对话模块测试用例**（5个）：问答、多轮对话、上下文理解等。

### 5.2.2 测试执行结果

功能测试通过率：43/44 = 97.7%。

未通过用例：研究空白挖掘在特定类型论文上的准确率略低于预期，已记录为已知问题。

## 5.3 性能测试

### 5.3.1 测试方案

使用Apache Bench进行压力测试，测试场景：文献上传、分析请求、检索查询。

测试指标：响应时间（平均、P95、P99）、吞吐量（QPS）、错误率、资源占用（CPU、内存）。

### 5.3.2 测试结果

**单用户场景**：文献上传平均响应时间2.3秒，分析任务提交平均响应时间0.8秒，检索查询平均响应时间0.3秒。

**50并发场景**：文献上传平均响应时间5.6秒，吞吐量8.9 QPS；分析任务提交平均响应时间2.1秒，吞吐量23.8 QPS；检索查询平均响应时间1.2秒，吞吐量41.5 QPS。

**资源占用**：CPU峰值占用65%，内存占用约4GB（包含Milvus向量缓存）。

## 5.4 测试结果分析

### 5.4.1 功能测试分析

系统核心功能基本正常，发现的问题主要是边界情况处理不完善，已修复。

研究空白挖掘模块在特定领域论文上的准确率有待提升，计划在后续版本中优化提示词。

### 5.4.2 性能测试分析

系统在50并发下表现良好，响应时间在可接受范围内。检索查询性能最优，受益于Milvus的高效向量索引。

文献上传和分析任务在并发较高时响应时间增加，符合预期。可通过增加Worker进程数进一步提升处理能力。

# 第6章 结论与展望

## 6.1 论文工作总结

本文设计并实现了一套基于人工智能的科研文献智能分析系统。系统采用B/S架构，集成了PDF解析、大语言模型、向量检索、知识图谱等先进技术，为科研人员提供了从文献管理到智能分析再到代码生成的完整解决方案。

**主要完成的工作**：

（1）完成了系统的需求分析，识别了8个功能模块，设计了44个功能测试用例。

（2）完成了系统的架构设计，采用分层架构和异步处理架构，设计了13张数据表和56个API接口。

（3）完成了系统的核心功能实现，包括PDF解析、AI分析、向量检索、知识图谱、AI对话、代码生成等模块。

（4）完成了系统的测试验证，功能测试通过率97.7%，性能测试满足设计要求。

**系统的主要特点**：

（1）私有化部署：支持本地部署，数据安全可控。

（2）中英文双语：针对中文文献和中文用户进行了专门优化。

（3）闭环设计：构建了从研究空白识别到代码生成的完整闭环。

（4）高性能：支持100并发，单篇分析时间<5分钟。

## 6.2 存在问题与不足

**存在的问题**：

（1）PDF解析对复杂版式（如多栏混排、图文环绕）的处理准确率有待提升。

（2）大模型API调用成本较高，需要进一步优化调用策略。

（3）知识图谱在大规模数据下的渲染性能有待优化。

（4）研究空白挖掘在特定领域上的准确率还有提升空间。

**不足之处**：

（1）系统测试覆盖度有待提升，部分边界情况处理不够完善。

（2）移动端适配尚未完成，目前主要支持桌面端访问。

（3）多用户协作功能尚未实现，暂不支持团队共享和协作。

## 6.3 未来工作展望

**短期计划**（3个月内）：

（3）实现大模型调用缓存机制，降低API成本。

（1）开发移动端适配版本，支持手机和平板访问。

（2）实现多用户协作功能，支持团队共享和权限管理。

（3）引入更先进的图谱渲染技术，支持大规模数据可视化。

**长期计划**（1年内）：

（1）支持更多文件格式（如Word、LaTeX）的解析和分析。

（2）引入多模态理解能力，支持图表内容的自动分析。

## 附录A 核心源代码说明

本附录对系统的核心源代码进行说明，详细代码见附件电子版。

### A.1 PDF解析模块核心代码

**文件**：src/pdf_parser_enhanced.py

**核心类**：PDFParser

**主要方法说明**：

- parse(file_path): 解析PDF文件，返回结构化数据
- extract_title(): 基于字体大小和位置特征提取标题
- extract_authors(): 基于位置关系和邮箱模式识别作者
- extract_abstract(): 基于关键词定位和段落提取识别摘要
- extract_tables(): 使用pdfplumber提取表格结构

**关键算法**：版式分类采用基于规则的方法，根据文本块的分布特征判断单栏/双栏版式。

### A.2 AI分析模块核心代码

**文件**：src/ai_analyzer.py

**核心类**：AIAnalyzer

**主要方法说明**：

- analyze(paper_content, analysis_type): 执行指定类型的分析
- generate_summary(): 生成论文摘要
- extract_keypoints(): 提取12类关键要点
- identify_gaps(): 识别研究空白

**提示词设计**：每种分析类型有专门的提示词模板，存储在src/prompts/目录下。

### A.3 向量检索模块核心代码

**文件**：src/vector_store.py

**核心类**：VectorStore

**主要方法说明**：

- add_paper(paper_id, text): 添加论文到向量库
- search(query, top_k=10): 语义检索相似论文
- delete(paper_id): 删除论文向量

**嵌入模型**：使用BGE-large-zh模型，通过sentence-transformers加载。

### A.4 知识图谱模块核心代码

**文件**：src/knowledge_graph.py、frontend/src/components/KnowledgeGraph.vue

**后端核心方法**：

- extract_entities(text): 基于LLM抽取实体
- identify_relations(entities): 识别实体间关系
- build_graph(papers): 构建知识图谱

**前端核心组件**：使用D3.js的forceSimulation实现力导向布局。

### A.5 AI对话模块核心代码

**文件**：src/chat_engine.py

**核心类**：ChatEngine

**主要方法说明**：

- chat(message, history): 处理用户消息，返回答复
- retrieve_context(query): 向量检索相关论文片段
- format_prompt(query, context, history): 组装提示词

**RAG实现**：检索+生成的两阶段架构，支持上下文记忆。

### A.6 代码生成模块核心代码
